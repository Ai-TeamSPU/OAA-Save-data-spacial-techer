# วิเคราะห์ความปลอดภัยและแนวทางการปรับปรุงสถาปัตยกรรมระบบ OAA
*(ประเด็นการแนบไฟล์เอกสาร, กฎหมาย PDPA และการแยกระบบ Frontend & Backend)*

เอกสารฉบับนี้วิเคราะห์ระดับความปลอดภัยของระบบบันทึกข้อมูลอาจารย์พิเศษ (OAA) ในปัจจุบัน ข้อควรระวังด้านข้อมูลส่วนบุคคล (PDPA) เมื่อต้องการเพิ่มฟังก์ชันแนบไฟล์ และการพิจารณาแยกโครงสร้าง Frontend / Backend

---

## 1. วิเคราะห์ความปลอดภัยของระบบ OAA ในปัจจุบัน (Security Assessment)

ระบบในปัจจุบันถูกออกแบบมาให้เป็นระบบกึ่งออฟไลน์ที่ใช้ **Babel Standalone** รัน React บนเบราว์เซอร์ ร่วมกับ **Google Sheets + Google Apps Script (GAS)** เป็นสเปรดชีตจัดเก็บข้อมูล ซึ่งหากประเมินด้านความปลอดภัยข้อมูลส่วนบุคคล (PDPA) ระบบปัจจุบันมี **ความเสี่ยงสูงมาก (High Risk)** ด้วยเหตุผลดังต่อไปนี้:

### 🔴 ความเสี่ยงด้านการเปิดเผยข้อมูล (Data Exposure Risks)
1. **สิทธิ์การเข้าถึงแบบสาธารณะของ Apps Script (Anyone):**
   - เพื่อให้ผู้ใช้งานทั่วไปสามารถกดส่งข้อมูลได้ โค้ดฝั่งหลังบ้าน [code.gs](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/OAA โปรแกรมบันทึกอาจารย์พิเศษ/google-apps-script/code.gs#L9) จึงต้องตั้งค่า Deploy เป็น `Who has access: Anyone`
   - เมื่อตรวจสอบโค้ดในฟังก์ชัน `doGet` พบว่ามีคำสั่ง:
     ```javascript
     if (e.parameter.action === "read") { return getRecords(); }
     ```
     นั่นหมายความว่า หากมีใครก็ตามล่วงรู้หรือค้นเจอ **Web App URL** ของ Apps Script นี้ จะสามารถเรียกคำสั่ง `?action=read` เพื่อดึงข้อมูลประวัติอาจารย์พิเศษทั้งหมด (รวมถึง ชื่อ, เบอร์โทร, อีเมล, ประวัติการทำงาน) ออกไปเป็น JSON ได้ทันทีโดยไม่ต้องเข้าสู่ระบบ
2. **การรั่วไหลผ่าน Local Storage:**
   - ข้อมูลประวัติอาจารย์ที่ถูกบันทึกไว้ในเครื่องจะเก็บใน `localStorage` ของเบราว์เซอร์ในรูปแบบข้อความดิบ (Plain text) ที่ไม่ได้เข้ารหัส ซึ่งผู้อื่นที่มาใช้คอมพิวเตอร์เครื่องเดียวกัน หรือมัลแวร์ในเบราว์เซอร์สามารถดึงข้อมูลออกไปได้ง่าย
3. **การเก็บรหัสผ่าน Admin แบบ Hard-coded:**
   - รหัสผ่าน Admin สำหรับเปิดการตั้งค่า Google Sheets ถูกเขียนไว้ในโค้ดดิบโดยตรง [app.jsx:1263](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/OAA โปรแกรมบันทึกอาจารย์พิเศษ/app.jsx#L1263) (`result.value === "123456"`) ทำให้ผู้ใช้งานที่มีทักษะไอทีสามารถเปิดดูรหัสผ่านได้ผ่านทาง Developer Tools ของเบราว์เซอร์

### ⚠️ ข้อจำกัดทางเทคนิคในการส่งไฟล์แนบผ่าน Google Sheets
- ปัจจุบันระบบใช้ HTTP GET ส่งข้อมูลไปใน URL Query String (`?data=...`) ซึ่งมีข้อจำกัดเรื่องขนาดข้อมูล (ไม่เกิน 2KB - 8KB) 
- หากต้องการเพิ่มฟังก์ชันแนบไฟล์ประเภทรูปภาพหรือเอกสาร PDF (มีขนาดหลัก MB) **จะไม่สามารถใช้วิธีส่งข้อมูลผ่าน URL แบบเดิมได้** จะต้องใช้การอัปโหลดไฟล์ผ่านช่องทาง POST ไปยังคลาวด์จัดเก็บไฟล์โดยตรง (เช่น Google Drive, AWS S3)

## 2. ผลกระทบด้าน PDPA เมื่อมีระบบแนบไฟล์ส่วนบุคคล (PDPA Implications)

เอกสารข้อมูลส่วนตัว เช่น **สำเนาบัตรประชาชน, สำเนาทะเบียนบ้าน, เอกสารวุฒิการศึกษา (Transcript), หรือใบรับรองเงินเดือน** จัดเป็นข้อมูลที่มีความอ่อนไหวสูงตามกฎหมาย PDPA หากระบบบันทึกและจัดเก็บเอกสารเหล่านี้โดยตรง จะต้องดำเนินการตามข้อกำหนดความปลอดภัยดังนี้:

1. **การควบคุมสิทธิ์การเข้าถึง (Access Control):** เอกสารที่อัปโหลดต้องเป็นแบบ **Private** ห้ามเข้าถึงได้จาก URL สาธารณะ และต้องมีระบบยืนยันตัวตน (Authentication) เฉพาะผู้ดูแลระบบที่ได้รับอนุญาตเท่านั้นที่จะกดดาวน์โหลดไฟล์มาดูได้
2. **ระบบสืบค้นประวัติการเปิดดู (Audit Log):** ระบบต้องสามารถบันทึกได้ว่า ใครเป็นคนดาวน์โหลดไฟล์ของอาจารย์ท่านใดไปดู และดูเมื่อไหร่
3. **การทำลายข้อมูลตามระยะเวลา (Data Retention Policy):** หากอาจารย์ท่านนั้นสิ้นสุดกระบวนการสมัครหรือไม่ได้รับการคัดเลือก เอกสารส่วนตัวต้องได้รับการลบทำลายอย่างถาวรจากฐานข้อมูลและถังขยะคลาวด์

> [!WARNING]
> **การใช้ Google Drive สาธารณะในการรับไฟล์อัปโหลดมีความเสี่ยงสูง:** 
> หากใช้ Apps Script รับอัปโหลดไฟล์ไปที่ Google Drive แล้วใช้วิธีเปลี่ยนสิทธิ์ลิงก์เป็นสาธารณะ (Anyone with link can view) เพื่อให้แอดมินกดเปิดจากสเปรดชีต ลิงก์ดังกล่าวอาจหลุดออกสู่ภายนอก และทำให้ผู้ไม่มีส่วนเกี่ยวข้องเข้าถึงเอกสารสำคัญของอาจารย์ได้

## 3. ควรแยกฝั่ง Frontend และ Backend หรือไม่? ต้องรื้อระบบใหม่ทั้งหมดไหม?

**คำแนะนำ:** **สมควรแยกสถาปัตยกรรมเป็น Frontend และ Backend อย่างแท้จริงครับ** หากระบบ OAA นี้จะใช้บันทึกข้อมูลและรับไฟล์แนบที่เป็นความลับจริง

อย่างไรก็ตาม **ไม่จำเป็นต้องรื้อเขียนโค้ดใหม่ทั้งหมด** เนื่องจากหน้าตาและตรรกะในฝั่ง Frontend ปัจจุบันสามารถยกมาใช้ต่อได้เกือบ 90% โดยปรับปรุงเฉพาะในส่วนของการส่งข้อมูลและส่วนของการจัดการไฟล์แนบ

เปรียบเทียบโครงสร้างการปรับปรุงระบบดังนี้:

### โครงสร้างและแนวทางการปรับปรุงระบบ

## 4. แผนงานการปรับเปลี่ยนระบบ (System Migration Steps)

หากต้องการปรับปรุงระบบให้ปลอดภัยตามมาตรฐาน PDPA และรองรับการแนบไฟล์หลายไฟล์ แนะนำให้ใช้กระบวนการปรับเปลี่ยนแบบเป็นขั้นตอน (Step-by-step) ดังนี้เพื่อหลีกเลี่ยงการรื้อโปรเจคทั้งหมด:

### ขั้นตอนที่ 1: การพัฒนา Frontend เพิ่มเติม (ใช้โค้ดเดิม 90%)
- **ใช้โค้ดเดิม:** นำไฟล์ [components.jsx](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/OAA โปรแกรมบันทึกอาจารย์พิเศษ/components.jsx) และ [app.jsx](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/OAA โปรแกรมบันทึกอาจารย์พิเศษ/app.jsx) มาใช้งานต่อได้เลย
- **ส่วนที่เพิ่ม/แก้ไขในโค้ดเดิม:**
  1. เพิ่มฟิลด์อัปโหลดไฟล์ด้วย React (เช่น การใช้ `<input type="file" multiple />` ในฝั่ง UI)
  2. เปลี่ยนจากฟังก์ชัน `fetch` ส่งข้อมูลไปยัง Google Sheets แบบเดิม [app.jsx:1301](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/OAA%20%E0%B9%82%E0%B8%9B%E0%B8%A3%E0%B9%81%E0%B8%81%E0%B8%A3%E0%B8%A1%E0%B8%9A%E0%B8%B1%E0%B8%99%E0%B8%97%E0%B8%B6%E0%B8%81%E0%B8%AD%E0%B8%B2%E0%B8%88%E0%B8%B2%E0%B8%A3%E0%B8%A2%E0%B9%8C%E0%B8%9E%E0%B8%B4%E0%B9%80%E0%B8%A8%E0%B8%A9/app.jsx#L1301) ให้ยิงข้อมูลเป็น `FormData` (รองรับข้อความร่วมกับไฟล์ไบนารี) ไปยัง Backend Server ตัวใหม่

### ขั้นตอนที่ 2: พัฒนา Backend Server ตัวใหม่ (เขียนเพิ่มขึ้นมาใหม่)
- เลือกใช้ Node.js (Express), Python (FastAPI/Django) หรือภาษาที่มหาวิทยาลัยศรีปทุมใช้งานอยู่เป็นหลัก
- **หน้าที่ของ Backend:**
  1. **รับข้อมูล (Endpoint POST /api/register):** ทำหน้าที่รับข้อความตัวอักษรและตรวจสอบไฟล์แนบ (จำกัดนามสกุลไฟล์เช่น .pdf, .jpg และขนาดไฟล์ไม่เกิน 5MB)
  2. **บันทึกข้อมูลลงฐานข้อมูล:** จัดเก็บข้อมูลลงฐานข้อมูลที่มีการป้องกันและเข้ารหัสข้อมูลส่วนบุคคล (เช่น PostgreSQL/MySQL)
  3. **บันทึกไฟล์ (Secure Storage):** อัปโหลดไฟล์เอกสารไปยัง Object Storage และบันทึกเฉพาะที่อยู่ไฟล์ (File path) ลงในฐานข้อมูล
  4. **ระบบความปลอดภัย (Admin Portal):** พัฒนาระบบ Login แยกต่างหากสำหรับเจ้าหน้าที่ เพื่อเรียกดูข้อมูลและไฟล์แนบ ซึ่งจะใช้การสร้าง **Pre-signed URL** ที่มีอายุการเข้าถึงจำกัด (เช่น ใช้ได้แค่ 5 นาทีหลังจากกดเปิดดู) เพื่อป้องกันการนำ URL ไปแชร์ต่อภายนอก
